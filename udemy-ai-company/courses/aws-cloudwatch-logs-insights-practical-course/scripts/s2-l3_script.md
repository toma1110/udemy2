# Section 2 Lecture 3 台本

### Title

parseとrequestId追跡

## Slide 1

### Slide Title

追跡する

### Slide Message

1つのリクエストを時系列で見る

### Narration

このレクチャーでは、パースとリクエストアイディー追跡を扱います。集計で問題の時間帯や種類が見えたら、次は1つのリクエストをじけいれつで追います。件数の増加は全体像を教えてくれますが、どの処理で失敗したかは、個別ログを読まないと分かりません。集計から個別追跡へ移ることで、原因の候補を具体化できます。ここからは、ログを広く集める段階から、1つの事例を深く読む段階へ切り替わります。 追跡の目的は、一つのリクエストがどこで遅くなり、どこで失敗したかをじけいれつで読むことです。集計で全体像を見たあと、代表例を一件選んで深掘りします。

### Visual Notes

- Timeline for one request across services
- requestId as thread

## Slide 2

### Slide Title

JSONログ

### Slide Message

フィールドがあればそのまま使う

### Narration

ジェイソンログでは、サービス名、ステータスコード、処理時間、リクエストアイディーなどがフィールドとして見えることがあります。この場合は、無理にパースせず、そのフィールドをそのまま使うのが簡単です。まずフィールズで見たい列を選び、フィルターで対象のリクエストアイディーやエラータイプに絞ります。構造化ログは、調査を速くするための土台です。 ジェイソンログは、フィールド名が安定していれば強力です。アプリ側でリクエストアイディーやユーザー識別子を出しておくと、調査時に検索条件として使えます。

### Visual Notes

- JSON log fields auto-discovered
- requestId, service, durationMs

## Slide 3

### Slide Title

parse

### Slide Message

文字列から値を取り出す

### Narration

非構造ログでは、メッセージの中から必要なあたいを取り出すことがあります。そこで使うのがパースです。ワイルドカードや正規表現で、ユーザー、処理名、エラー理由、リクエストアイディーなどをフィールドとして取り出します。ただし、パースはログ形式に強く依存します。まずサンプルを数行見て、変わらない文字列と変わるあたいを分けてから作ります。 パースは、既存ログをすぐ構造化ログへ変えられない時の橋渡しです。ただし、ログ形式が変わると抽出できなくなるため、重要な項目は将来的に構造化して出す設計が望ましいです。

### Visual Notes

- Raw message transformed into fields
- parse command as extractor

## Slide 4

### Slide Title

requestId

### Slide Message

同じキーで出来事をつなぐ

### Narration

リクエストアイディーは、一連の処理をつなぐキーです。同じリクエストアイディーで絞ると、受付、認証、下流呼び出し、例外、終了ログをじけいれつで見られます。複数サービスをまたぐ場合は、リクエストアイディー、トレースアイディー、コリレーションアイディーなど名前が違うこともあります。重要なのは、同じユーザー操作を追える共通キーをログに残しておくことです。共通キーがない場合、時刻、ユーザー、パス、処理名を組み合わせて近いログを探すことになりますが、精度は落ちます。

### Visual Notes

- Same requestId connecting multiple log events
- Start, service, error, end

## Slide 5

### Slide Title

dedup

### Slide Message

同じ対象の重複を抑える

### Narration

ディーダップは、同じ対象の重複を抑える時に使います。たとえば同じリクエストアイディーで複数行出ている時に、最後の状態だけを見たい場合があります。ただし、ディーダップすると途中経過が見えなくなることもあります。原因調査では、まずじけいれつで全部見て、一覧や集計に戻る時に重複を抑える、という使い分けが安全です。

### Visual Notes

- Duplicate request rows collapsed
- dedup requestId

## Slide 6

### Slide Title

時系列で読む

### Slide Message

ascで古い順に並べる

### Narration

1つのリクエストを読む時は、古い順に並べるのが基本です。開始から終了までを順番に見ることで、どこまでは成功し、どこから失敗したかが分かります。新しい順のままだと、結果から原因へ逆向きに読まなければなりません。障害対応では、集計で全体を見て、リクエスト追跡で具体例を読む。この往復が、原因に近づくための実務的な流れです。最後に、個別追跡で見つけた原因候補をもう一度集計へ戻し、同じ事象が何件あるかを確認します。 古い順に並べると、入口、内部処理、外部呼び出し、エラー出力の順番が見えます。新しい順では分かりにくい因果関係を読みやすくするための並べ替えです。

### Visual Notes

- Ascending timeline
- Root cause path
